# CLI Definition Patterns

> How `opencli` defines commands, compiles them into the manifest, and executes
> them at runtime.

`opencli` supports two authoring modes:

- YAML pipeline adapters under `src/clis/<site>/<command>.yaml`
- TypeScript adapters that call `cli({...})` from `src/clis/<site>/<command>.ts`

Both shapes end up as `CliCommand` records in the registry. The important code
paths are:

- `src/registry.ts:7-93` for `Strategy`, `Arg`, `CliCommand`, and `cli()`
- `src/engine.ts:25-202` for discovery, manifest loading, YAML parsing, and
  runtime dispatch
- `src/build-manifest.ts:21-198` for build-time manifest generation
- `src/pipeline/executor.ts:13-93` for YAML pipeline execution
- `src/pipeline/template.ts:5-159` for `${{ ... }}` interpolation

## Pre-Development Checklist

Read these files before adding or changing command definitions:

1. `src/registry.ts`
   - Understand `Strategy`, `CliOptions`, `CliCommand`, and `cli()` defaults.
2. `src/engine.ts`
   - Understand how YAML files are registered in development and how manifest
     entries are loaded in builds.
3. `src/build-manifest.ts`
   - Required for TypeScript adapters. Manifest extraction is regex-based and
     expects literal metadata.
4. `src/pipeline/executor.ts`
   - Required for YAML adapters. It defines the runtime-supported step set.
5. `src/pipeline/template.ts`
   - Required whenever you use `${{ ... }}` interpolation or embedded
     `evaluate` scripts.
6. Representative adapters
   - YAML: `src/clis/hackernews/top.yaml`, `src/clis/twitter/trending.yaml`,
     `src/clis/xiaohongshu/feed.yaml`
   - TypeScript: `src/clis/twitter/search.ts`, `src/clis/bilibili/search.ts`,
     `src/clis/twitter/post.ts`

## Repo-Specific Notes

- `strategy` is important metadata, but `main.ts` branches on `cmd.browser`
  when deciding whether to create a browser session (`src/main.ts:150-152`).
- Built artifacts rely on `dist/cli-manifest.json`, generated by
  `npm run build` (`package.json:16-28` and `src/build-manifest.ts:170-198`).
- The validator recognizes a `scroll` step name (`src/validate.ts:6-12`), but
  the executor does not register a `scroll` handler
  (`src/pipeline/executor.ts:21-38`). Treat `scroll` as unsupported until the
  runtime grows a real step implementation.
